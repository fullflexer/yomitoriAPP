# yomitoriAPP 要求定義書

## 概要
yomitoriAPP は、戸籍謄本・除籍謄本・改製原戸籍等を OCR / 画像解析で読み取り、人物・身分事項・続柄を構造化データへ変換し、法定相続人の候補判定と「相続関係説明図」の自動生成を支援する Web アプリケーションである。主要ユーザーは司法書士・行政書士であり、一般個人および士業事務所スタッフも利用対象とする。

本書は、日本国内の相続法実務を前提に、MVP から本番展開までの要求を定義する。法令・実務運用・審査実務が未確認の事項は断定せず、`要確認` として OQ を付す。

### MVP スコープ
- 包含: コンピュータ化戸籍(H6年式以降横書き活字) / 核家族(被相続人+配偶者+子) / 法定相続(遺言なし) / 単一被相続人
- 除外: 手書き戸籍(昭和改製前) / 改製原戸籍 / 数次相続 / 養子の実親関係判定 / 外国籍 / 遺言分割 / 相続税計算 / 複数被相続人の同時処理

### 前提・法令適用方針
- 準拠法は日本法とし、相続判定の根拠は主として民法第882条、第886条、第887条、第889条、第890条、第891条、第892条ないし第895条、第900条、第901条、第915条、第938条、第30条ないし第32条、養子関係は民法第727条および特別養子縁組規定を参照する。
- 戸籍情報の取得・利用設計は戸籍法の証明書請求関連規定（第10条系統）および戸籍法施行規則第73条、第79条の6、第79条の9等の関連規定を前提とし、証明書等の広域交付については法務省公表のとおり 2024年3月1日施行の制度を考慮する。ユーザー指定の「令和6年4月1日施行」については、公式公表との差異があるため、本書では 2024年3月1日を採用する。
- 戸籍情報は法的に機微性の高い情報であり、本サービスでは社内分類上も最高機密相当として扱う。

## 1. 入力・OCR要件

### REQ-001 対応書類と入力ゲーティング
- 要件ID: REQ-001
- 優先度: Must
- 要件本文: MVP ではコンピュータ化戸籍(H6年式以降横書き活字)の戸籍謄本・除籍謄本を正式対応対象とし、アップロード時に書類種別、帳票様式、印字/手書き、解像度、ページ向き、欠損の有無を自動判定して処理可否を振り分ける。市区町村ごとのフォーマット差異には、帳票プロファイル管理、欄見出し辞書、抽出後正規化ルールで対処する。改製原戸籍、非コンピュータ化戸籍、昭和改製前の手書き戸籍は自動判定対象から外し、要確認事項は [OQ-4](./open-questions.md#oq-4) を参照して将来フェーズ送りとして扱う。
- 背景: OCR 精度と相続判定精度を初期フェーズで担保するには、帳票ばらつきが比較的小さい活字・コンピュータ化戸籍に入力を絞る必要がある。
- 受入基準: 対応対象・非対応対象が UI 上で明示され、非対応文書は処理開始前にブロックまたは要レビュー案件として隔離される。
- 依存: なし

### REQ-002 画像前処理と最低品質要件
- 要件ID: REQ-002
- 優先度: Must
- 要件本文: OCR 前に傾き補正、トリミング、ノイズ除去、コントラスト補正、ページ分離、向き推定、欄外除去を行い、最低品質要件として 300dpi 相当以上、主要欄が欠けないこと、強い影・折れ・ぼけが主要欄を覆わないことを満たさない画像は再撮影または再スキャンを要求する。
- 背景: 戸籍は氏名・続柄・生年月日・身分事項の誤読が後段の名寄せと相続判定に連鎖するため、前処理段階で品質を揃える必要がある。
- 受入基準: 画像診断で品質不足が検知された場合、再提出理由が具体的に表示され、品質基準を満たした画像のみ OCR 実行対象となる。
- 依存: REQ-001

### REQ-003 OCR パイプラインと人間補正フロー
- 要件ID: REQ-003
- 優先度: Must
- 要件本文: OCR は「画像前処理 -> マルチモーダル OCR/画像理解 -> 項目抽出 -> ルールベース整合性検証 -> 人間補正 -> 確定」の段階処理とし、LLM/マルチモーダルモデルはレイアウト理解と候補抽出、ルールベースは日付形式・続柄語彙・戸籍記載パターン・必須項目検証を担当する。低信頼フィールド、同一人物同定に影響するフィールド、死亡・婚姻・認知・養子関連イベントは必ず人間確認を経る。
- 背景: 戸籍 OCR は自由文ではなく半構造帳票であり、モデル単独では説明可能性が不足するため、人間補正とルール検証の併用が必要である。
- 受入基準: 各抽出フィールドに信頼度・抽出根拠・要確認フラグが付与され、補正後の確定値と原画像位置の対応が追跡できる。
- 依存: REQ-001, REQ-002

### REQ-004 OCR 精度測定方法
- 要件ID: REQ-004
- 優先度: Must
- 要件本文: OCR 精度はゴールドスタンダードデータセットで測定する。データセットは実戸籍を使用せず、法務省様式ベースの合成データを 100 枚以上作成し、様式別・品質別・手書き/活字別に層化サンプリングする。CER（Character Error Rate）は「置換数 + 挿入数 + 削除数」を正解文字数で除した値とし、用途別目標は氏名 CER ≤2%、住所 CER ≤5% とする。加えてフィールド正解率を「対象フィールドの完全一致件数 / 総件数」と定義し、主要フィールドの合格基準を 90%以上とする。最終目標値は要確認事項として [OQ-4](./open-questions.md#oq-4) を参照する。
- 背景: 実戸籍を評価用に広く使うことは法的・倫理的に難しく、再現可能な性能評価には合成データを用いた測定基盤が必要である。
- 受入基準: ゴールドスタンダード、CER、フィールド正解率の定義が評価手順書に明記され、氏名・住所・続柄・出生/死亡日の計測結果が継続比較できる。
- 依存: REQ-001, REQ-003

## 2. データモデル要件

### REQ-101 Person エンティティ定義
- 要件ID: REQ-101
- 優先度: Must
- 要件本文: Person エンティティは、氏名、氏名正規化表現、旧字体/異体字表現、性別記載、生年月日、死亡日、出生地、本籍、最後の住所、登記簿上住所、戸籍上の役割、備考、原文スナップショット、ソース文書参照を保持できること。
- 背景: 相続実務では同一人物であっても、転籍、改名、旧字体、住所差異が多く、正規化値と原文値の両方を保持しないと説明責任を果たせない。
- 受入基準: Person が原文値・正規化値・出典情報を同時保持し、後段の名寄せと出力生成で利用できる。
- 依存: REQ-003

### REQ-102 Event / Relationship / 時系列管理
- 要件ID: REQ-102
- 優先度: Must
- 要件本文: Event エンティティは出生、認知、婚姻、離婚、養子縁組、離縁、転籍、分籍、死亡、失踪宣告、相続放棄、廃除、欠格関連の審査記録を扱い、Relationship は実親子、養親子、配偶者、前配偶者、兄弟姉妹、半血兄弟姉妹、同一戸籍所属を表現できること。各イベントは発生日、効力発生日、出典ページ、確実性区分を持つ。
- 背景: 相続判定は現在の家族関係だけでなく、時点ごとの法的身分変動を追跡する必要がある。
- 受入基準: 婚姻 -> 離婚 -> 再婚、養子縁組 -> 離縁などの連続イベントが時系列に保存され、任意時点の家族関係が再構成できる。
- 依存: REQ-101

### REQ-103 名寄せと戸籍連続性検証
- 要件ID: REQ-103
- 優先度: Must
- 要件本文: 複数戸籍間の同一人物同定は、氏名、異体字正規化、生年月日、続柄、本籍移転履歴、配偶者/親子リンク、死亡情報の一致度を用いたスコアリングで行い、出生から死亡までの戸籍連続性が途切れる場合は自動確定せず要レビューにする。
- 背景: 戸籍は転籍・改製・除籍をまたいで連なるため、単票 OCR の正しさだけでは相続判定の正しさを担保できない。
- 受入基準: 自動名寄せの候補一覧、確信度、差分理由、連続性警告が UI と監査ログに出力される。
- 依存: REQ-101, REQ-102

### REQ-104 不確実性・出典・監査可能性
- 要件ID: REQ-104
- 優先度: Must
- 要件本文: すべての抽出値と関係推定は「確定 / 候補 / 保留 / 不一致」の状態を持ち、元画像座標、抽出モデル、抽出時刻、補正者、補正理由、比較前後値を保持する。
- 背景: 戸籍 OCR と相続判定は完全自動化よりも再現性と監査性が重要であり、判断過程を再構築できなければ士業実務に耐えない。
- 受入基準: 任意の人物・イベント・関係について、どの文書のどの位置から抽出されたかを追跡できる。
- 依存: REQ-003, REQ-103

## 3. 相続判定ロジック要件

### REQ-201 法定相続人確定フロー
- 要件ID: REQ-201
- 優先度: Must
- 要件本文: 相続判定は民法第882条、第887条、第889条、第890条、第900条を基礎に、被相続人死亡時点での配偶者有無、子・直系卑属の有無、直系尊属の有無、兄弟姉妹の有無を順に判定し、法定相続人候補と順位を機械可読な形で出力する。
- 背景: 相続人判定の中核は順位判定であり、ここが不透明だと後続の相続分計算や図面生成が成立しない。
- 受入基準: 配偶者 + 子、配偶者 + 直系尊属、配偶者 + 兄弟姉妹、子のみ等の基本ケースで順位と候補者一覧が正しく出力される。
- 依存: REQ-102, REQ-103

### REQ-202 代襲相続・養子・半血兄弟・胎児・失踪宣告
- 要件ID: REQ-202
- 優先度: Must
- 要件本文: 代襲相続・再代襲相続、普通養子・特別養子、半血兄弟姉妹、胎児、失踪宣告は別ルールとして実装する。代襲は民法第887条および第889条系統、法定相続分は第900条・第901条、失踪宣告は第30条ないし第32条、養子関係は第727条および特別養子縁組規定を参照する。MVP では養子の実親関係判定を除外し、戸籍記載だけで法的効果を確定できない案件は自動確定せず法務レビュー待ちとする。
- 背景: これらの例外は頻度は低くても、判定を誤ると法的効果が大きい。
- 受入基準: 代襲発生条件、半血兄弟の相続分差、胎児を既に生まれたものとみなす扱い、失踪宣告による死亡みなし時点がルールとして表現され、説明文が生成できる。
- 依存: REQ-201

### REQ-203 相続放棄・欠格・廃除・数次相続・判定不能条件
- 要件ID: REQ-203
- 優先度: Must
- 要件本文: 相続放棄、相続欠格、推定相続人の廃除、数次相続、外国籍事案、複数被相続人同時処理は、MVP では自動確定対象にしない。家庭裁判所審判書、申述受理証明書等の外部証憑が確認できない限り、案件状態を「判定停止」または「要専門家レビュー」にする。相続放棄の期間・方式は民法第915条、第938条を参照し、責任分界は要確認事項として [OQ-2](./open-questions.md#oq-2) を参照する。
- 背景: 戸籍だけでは判定できない法的事実が多く、誤判定のリスクが高い。
- 受入基準: 上記ケースが検知されたとき、自動確定せず停止理由・必要書類・レビューロールが表示される。
- 依存: REQ-201, REQ-202

### REQ-204 法定相続分計算と説明可能性
- 要件ID: REQ-204
- 優先度: Should
- 要件本文: 法定相続分は民法第900条を基本に自動計算し、配偶者と各順位相続人の組合せごとに、数式、根拠条文、代襲考慮の有無、放棄・欠格除外の有無を表示する。判定不能または要確認条件が残る場合、割合は確定値ではなく参考値として表示し、出力物に「要確認」を付す。
- 背景: 実務では結論だけでなく、なぜその割合になったかの説明が必要である。
- 受入基準: 相続分表示画面または帳票に、候補者ごとの割合と根拠説明が出力される。
- 依存: REQ-201, REQ-202, REQ-203

## 4. 出力・図面要件

### REQ-301 相続関係説明図の必須記載事項
- 要件ID: REQ-301
- 優先度: Must
- 要件本文: 相続関係説明図には、被相続人の氏名、死亡日、最後の本籍、最後の住所、必要に応じた登記簿上住所、各相続人の氏名、住所、生年月日、被相続人との続柄、死亡・代襲・放棄等の注記を表示する。法務局提出用の最終体裁は全国一律の保証をせず、要確認事項として [OQ-5](./open-questions.md#oq-5) を参照する。
- 背景: 登記手続では被相続人・相続人・続柄・住所等の識別情報が不足すると補正対象になりやすい。
- 受入基準: 生成帳票に必須項目が欠落した場合は出力できず、欠落項目一覧が表示される。
- 依存: REQ-101, REQ-201

### REQ-302 レイアウトアルゴリズム
- 要件ID: REQ-302
- 優先度: Must
- 要件本文: 家系図ツリーは世代ごとの段組み、配偶者の横並び配置、親子の縦連結、死亡済み・代襲・離婚再婚・養子関係の注記衝突回避を満たす自動レイアウトを採用し、複雑ケースでノード重なりや線交差が過度に増えないことを要件とする。
- 背景: 相続関係説明図は情報密度が高く、レイアウト破綻がそのまま読み違いにつながる。
- 受入基準: 核家族ケースで A4 縦に収まり、複雑ケースでもノード重なりなし・主要関係線が視認可能な出力が得られる。
- 依存: REQ-301

### REQ-303 出力形式と手動微調整
- 要件ID: REQ-303
- 優先度: Should
- 要件本文: 出力形式は PDF、印刷用 A4/A3、画面表示用 SVG を正式対応とし、士業向けモードではノード位置、注記位置、改行、表示/非表示項目の手動微調整を可能にする。Canvas 出力は将来拡張とする。
- 背景: 法務局提出前に担当者が読みやすさを微調整したい需要が高い。
- 受入基準: 自動生成後にレイアウト微修正して再出力でき、修正履歴が案件バージョンとして残る。
- 依存: REQ-302

### REQ-304 自動生成表示と根拠メタデータ
- 要件ID: REQ-304
- 優先度: Must
- 要件本文: 生成帳票および画面には「自動生成」「要確認」「最終確認者」「生成日時」「対象案件ID」を表示し、必要に応じて根拠文書一覧と未確定論点を付録または別紙で出力できること。
- 背景: 自動生成物をそのまま確定資料と誤認させない表示が必要である。
- 受入基準: 低信頼フィールドや未確定判定が残る場合、帳票上に明示ラベルが出る。
- 依存: REQ-104, REQ-204

## 5. UX・業務フロー要件

### REQ-401 業務フロー定義
- 要件ID: REQ-401
- 優先度: Must
- 要件本文: 標準ワークフローは「案件作成 -> 書類アップロード -> 画像品質判定 -> OCR/構造化抽出 -> 確認/補正 -> 相続判定 -> 図面生成 -> レビュー承認 -> 出力」とし、各段階で案件状態を遷移管理する。
- 背景: 士業実務では誰がどの段階で何を確認したかが重要であり、単発アップロード型 UX では業務に乗らない。
- 受入基準: 案件一覧から現在状態、担当者、差戻し理由、次アクションが確認できる。
- 依存: REQ-002, REQ-003, REQ-201, REQ-301

### REQ-402 確認・補正 UI
- 要件ID: REQ-402
- 優先度: Must
- 要件本文: 補正 UI は原本画像と抽出結果の対照表示、信頼度表示、差分ハイライト、該当画像座標へのジャンプ、候補値選択、手入力上書き、補正理由入力を提供する。
- 背景: OCR の誤読修正は帳票と抽出値を往復しながら行うため、対照性が UX の中核となる。
- 受入基準: 氏名・日付・続柄・住所など主要フィールドについて、原画像の該当位置を 1 クリックで確認できる。
- 依存: REQ-003, REQ-104

### REQ-403 案件管理・保存・再編集
- 要件ID: REQ-403
- 優先度: Must
- 要件本文: 複数案件を保存・再編集可能とし、案件ごとに文書版、抽出版、判定版、図面版をバージョン管理する。状態は少なくとも「下書き」「OCR完了」「補正待ち」「レビュー待ち」「顧客確認待ち」「出力確定」を持つ。
- 背景: 相続案件は資料収集が段階的であり、1 回で完結しない。
- 受入基準: 過去バージョンの復元、比較、再出力が可能である。
- 依存: REQ-401

### REQ-404 ユーザー種別別 UX と監査ログ
- 要件ID: REQ-404
- 優先度: Should
- 要件本文: 司法書士・行政書士向けには案件一覧、レビュー待ちキュー、差戻し、監査ログ閲覧、手動微調整を提供し、一般個人向けには案内文・用語解説・入力支援を強める。すべての重要操作は監査ログに記録する。
- 背景: 主要ユーザーは士業である一方、一般個人にも理解可能な導線が必要である。
- 受入基準: 権限に応じて表示機能が切り替わり、監査ログから主要操作履歴を確認できる。
- 依存: REQ-403, REQ-604

## 6. 技術アーキテクチャ要件

### REQ-501 アプリケーション基盤
- 要件ID: REQ-501
- 優先度: Must
- 要件本文: フロントエンドは Next.js 16、型安全な API 層は tRPC、認可・業務ロジックは TypeScript を基本とし、ファイルストレージは R2、非同期ジョブはキュー駆動、生成物と中間データは案件単位で関連付ける。
- 背景: OCR、図面生成、監査ログは非同期処理と型安全性の両立が必要である。
- 受入基準: アップロード、OCR ジョブ、補正、図面生成、出力保存が案件単位で一貫管理できる設計になっている。
- 依存: REQ-401, REQ-403

### REQ-502 OCR/LLM パイプライン構成
- 要件ID: REQ-502
- 優先度: Must
- 要件本文: OCR パイプラインは「前処理 -> 画像理解/OCR -> 後処理 -> 構造化パース -> 整合性検証 -> 名寄せ -> 相続判定 -> 図面生成」で構成し、LLM 活用箇所は帳票理解、構造化パース、異常候補抽出、説明文生成に限定する。法定相続人判定、必須項目検証、出力制御はルールベースで実装する。
- 背景: 判断責務の切り分けを明確にしないと、精度低下時の原因切り分けと監査が困難になる。
- 受入基準: 各段階の入出力スキーマと失敗時の再試行・人間介入ポイントが定義される。
- 依存: REQ-003, REQ-103, REQ-201

### REQ-503 DB・セキュリティ・監査
- 要件ID: REQ-503
- 優先度: Must
- 要件本文: データベースは PostgreSQL+JSONB を第一候補とし、Person/Event/Relationship の正規化テーブルと OCR 中間成果物・抽出メタデータの JSONB を併用する。保存時暗号化、通信時 TLS、RBAC、案件単位アクセス制御、監査ログ、鍵管理、将来のテナント分離を必須とする。
- 背景: 戸籍データは高機微情報であり、構造化データと半構造中間データの両立が必要である。
- 受入基準: 権限のない利用者は他案件を参照できず、主要操作とデータ閲覧は監査ログに残る。
- 依存: REQ-104, REQ-404

### REQ-504 保持期間・削除・クラウド配置
- 要件ID: REQ-504
- 優先度: Should
- 要件本文: 戸籍画像・OCR 中間データ・生成帳票の保持期間は案件契約種別ごとに設定し、期限到来後は自動削除または不可逆匿名化を行う。外部 API 利用時の保存有無、学習利用有無、国外移転有無は契約別に管理し、保持期間の最終値は要確認事項として [OQ-3](./open-questions.md#oq-3) を参照する。
- 背景: 実務上の保管ニーズと個人情報最小化のバランスを取る必要がある。
- 受入基準: 保持・削除ポリシーが案件単位で設定でき、削除実行ログが残る。
- 依存: REQ-503, REQ-602

## 7. 法的・コンプライアンス要件

### REQ-601 日本法準拠と戸籍法制約
- 要件ID: REQ-601
- 優先度: Must
- 要件本文: 本サービスの相続判定と証跡設計は日本の民法、戸籍法、戸籍法施行規則を基礎とし、戸籍情報の取得主体・利用目的・出力範囲がそれらに反しないよう制御する。戸籍証明書等の請求と証明書様式は戸籍法第10条系統および戸籍法施行規則第79条の6、第79条の9等を参照し、広域交付制度は 2024年3月1日施行の法務省公表内容を前提にする。コンピュータ化されていない一部戸籍は取得対象外となり得る点を UI に表示する。
- 背景: 入手可能性と処理可能性を混同すると、サービス説明と実務が乖離する。
- 受入基準: 利用規約、ヘルプ、案件 UI に対応範囲と法的制約が反映される。
- 依存: REQ-001

### REQ-602 個人情報保護・外部API委託管理
- 要件ID: REQ-602
- 優先度: Must
- 要件本文: 戸籍データの処理は個人情報保護法に従い、委託先・SaaS・外部 API との関係を「委託」「第三者提供」「外国にある第三者提供」の別で整理する。外部 API は学習不使用、保存制限、アクセスログ、契約上の秘密保持を満たすものに限定し、法的整理が未確定な場合は要確認事項として [OQ-1](./open-questions.md#oq-1) を参照する。
- 背景: 委託と第三者提供の整理を誤ると、説明義務と同意設計を誤るおそれがある。
- 受入基準: 利用する外部サービスごとに法的整理、保存方針、越境移転有無、契約根拠を台帳化できる。
- 依存: REQ-503, REQ-504

### REQ-603 免責・責任分界・レビュー責任
- 要件ID: REQ-603
- 優先度: Must
- 要件本文: 相続人判定・図面生成は専門家の最終確認を前提とし、利用規約・画面表示・出力物に「自動判定の補助ツールであり、法的判断や登記受理を保証しない」旨を明記する。誤判定時の責任分界、補償範囲、専門家レビュー責任は要確認事項として [OQ-2](./open-questions.md#oq-2) を参照し、契約・規約で明示する。
- 背景: 法的効果のある実務で AI を用いる以上、責任主体を曖昧にできない。
- 受入基準: 出力画面、PDF、利用規約、管理画面に責任分界とレビュー前提が反映される。
- 依存: REQ-203, REQ-304

### REQ-604 電子証跡・説明責任・監査
- 要件ID: REQ-604
- 優先度: Could
- 要件本文: 将来フェーズでは、重要版の図面出力やレビュー完了記録に電子署名またはタイムスタンプを付与できるようにし、AI 判断の入力・出力・補正履歴・モデルバージョン・レビュー者を監査証跡として保持する。
- 背景: すぐに必須ではないが、士業向け本番運用では「いつ、誰が、何を根拠に確定したか」の証跡価値が高い。
- 受入基準: 将来拡張可能な監査イベント設計が用意され、主要判断イベントの完全性が担保できる。
- 依存: REQ-104, REQ-404, REQ-603

## 8. 開発フェーズ分割

### REQ-701 PoC フェーズ
- 要件ID: REQ-701
- 優先度: Must
- 要件本文: PoC は MVP 包含範囲に限定し、コンピュータ化戸籍(H6年式以降横書き活字)の OCR、核家族の Person/Event/Relationship 抽出、基本的な相続人順位判定、簡易図面生成を検証する。品質評価指標は CER、フィールド正解率、補正工数、図面修正率とする。
- 背景: 最初に最も高い技術リスクである OCR と構造化抽出の成立性を潰す必要がある。
- 受入基準: 合成データで REQ-004 の暫定目標に到達し、核家族ケースで図面が自動生成できる。
- 依存: REQ-004, REQ-201, REQ-302

### REQ-702 β フェーズ
- 要件ID: REQ-702
- 優先度: Must
- 要件本文: β では案件管理、補正 UI、監査ログ、法定相続分表示、士業向けレビュー導線を追加し、実利用に近い操作性を検証する。市区町村フォーマット差や品質劣化画像への耐性を拡張する。
- 背景: OCR 単体が成立しても、業務フローに組み込めなければ利用されない。
- 受入基準: 複数案件管理、補正後再判定、PDF/SVG 出力、レビュー承認フローが一連で動作する。
- 依存: REQ-401, REQ-402, REQ-403, REQ-503

### REQ-703 士業限定運用フェーズ
- 要件ID: REQ-703
- 優先度: Should
- 要件本文: 士業限定運用では、実案件での補正工数、レビュー時間、図面修正率、要確認停止率を計測し、改製原戸籍や代襲相続など次フェーズ対象の優先順位を決める。外部 API 利用、保持期間、責任分界は契約条件に反映する。
- 背景: 実務適合性は机上要件ではなく、導入先の運用データで詰める必要がある。
- 受入基準: KPI が定義され、少数の士業事務所で継続利用できる品質・運用手順が整う。
- 依存: REQ-602, REQ-603, REQ-702

### REQ-704 本番展開フェーズ
- 要件ID: REQ-704
- 優先度: Should
- 要件本文: 本番展開は、PoC/β/士業限定運用の品質指標と法務確認を満たした後に行う。改製原戸籍、数次相続、養子の実親関係判定、複数被相続人処理などの拡張は、OQ と実運用課題を解消した後に段階投入する。
- 背景: 例外事案まで一度に広げると誤判定リスクとサポート負荷が急増する。
- 受入基準: 本番投入チェックリストに、法令確認、精度指標、セキュリティ、監査、削除ポリシー、責任分界、法務局提出運用確認が含まれる。
- 依存: REQ-703, REQ-504, REQ-603

## 補足
- 本書で「法務局提出にも使える」とは、提出候補資料のたたき台を自動生成できることを意味し、個別案件・管轄法務局での受理保証を意味しない。受理基準の最終確認は要確認事項として [OQ-5](./open-questions.md#oq-5) を参照する。
- 実戸籍を学習・評価用の汎用データセットとして再利用する運用は採用しない。評価は法務省様式ベースの合成データを原則とする。
