# yomitoriAPP ロードマップ

本書は [requirements-definition.md](/Users/chion/Projects/yomitoriapp/docs/requirements-definition.md) の `REQ-701`〜`REQ-704` を具体化し、[edge-cases.md](/Users/chion/Projects/yomitoriapp/docs/edge-cases.md) のエッジケース対応順序と [tech-risk-register.md](/Users/chion/Projects/yomitoriapp/docs/tech-risk-register.md) の高リスク緩和計画をフェーズ別に整理したものである。

- 対象アプリ: yomitoriAPP（戸籍OCR→相続関係説明図自動生成Webアプリ）
- 前提: PoC 合格前に β へ進まない。β 品質基準と法務・業務監修レビューゲート通過前に士業限定運用へ進まない。士業限定運用での品質指標達成前に本番展開へ進まない。
- 法令・実務運用で未確定の論点は `要確認` とし、`OQ-1`〜`OQ-5`、`REQ-601`〜`REQ-603`、`RISK-004`、`RISK-009`、`RISK-010` の解消状況を各フェーズゲートで確認する。

## フェーズ間依存関係

| 依存関係 | 必須条件 | 主参照 |
| --- | --- | --- |
| Phase 1 → Phase 2 | PoC 合格基準を満たし、`REQ-004` の OCR 評価と `REQ-701` の受入基準を満たすこと | `REQ-004`, `REQ-701`, `RISK-001`, `RISK-008` |
| Phase 2 → Phase 3 | β 品質基準達成に加え、法務・業務監修レビューゲートを通過すること | `REQ-702`, `REQ-602`, `REQ-603`, `RISK-004`, `RISK-006`, `RISK-007` |
| Phase 3 → Phase 4 | 士業限定運用の KPI 達成、契約・規約・プライバシー整備、運用監査の完了 | `REQ-703`, `REQ-503`, `REQ-504`, `REQ-603`, `RISK-005`, `RISK-006`, `RISK-009`, `RISK-010` |

## 共通品質評価指標

各フェーズで以下を計測し、次フェーズ移行判定に用いる。

| 指標 | 定義 | Phase 1 目安 | Phase 2 目安 | Phase 3 目安 | Phase 4 目安 |
| --- | --- | --- | --- | --- | --- |
| OCR 精度 | CER、フィールド正解率 | 氏名 CER ≤2%、住所 CER ≤5%、主要フィールド正解率 90%以上 | 帳票差分込みで Phase 1 同等以上を維持 | 実案件匿名化サンプルでも β 比で劣化しない | 本番運用で継続監視し、しきい値逸脱時は自動で `要確認` に送る |
| 相続判定正解率 | テストケースに対する判定一致率 | MVP内基本ケースで 95%以上 | 司法書士監修 50 件で 95%以上 | 士業限定運用案件で 97%以上 | 本番運用で継続監視し、法改正時は再評価 |
| 名寄せ精度 | Precision / Recall / F1 | 基礎データセットで F1 0.90 以上 | 市区町村差分込みで F1 0.93 以上 | 実案件で F1 0.95 以上 | 本番で継続監視し、誤統合は即是正 |
| 補正工数 | 1案件あたりの人手修正時間 | 60分以内を目安 | 30分以内を目安 | 15分以内を目安 | 10分以内を目安 |
| 図面修正率 | 生成後に手動修正が必要だった割合 | 50%以下 | 30%以下 | 15%以下 | 10%以下 |

上記のうち、法令・受理実務に依存する最終的なしきい値は `要確認` とし、`OQ-4` と `OQ-5` の解消結果を反映する。

## Phase 1: PoC（概念実証）

- 関連要件: `REQ-001`〜`REQ-004`, `REQ-101`〜`REQ-104`, `REQ-201`, `REQ-203`, `REQ-301`, `REQ-302`, `REQ-304`, `REQ-501`, `REQ-502`, `REQ-701`
- 期間の目安: 6〜8週間

### スコープ

- コンピュータ化戸籍（H6年式以降横書き活字）に限定した OCR パイプラインを実装する（`REQ-001`, `REQ-002`, `REQ-003`）。
- Person / Event / Relationship の最小データモデルと、核家族中心の名寄せ・家族グラフ生成を実装する（`REQ-101`〜`REQ-103`）。
- 基本的な法定相続人順位判定と簡易相続関係説明図自動生成を実装する（`REQ-201`, `REQ-301`, `REQ-302`）。
- 非対応文書、MVP外ケース、低信頼フィールドを検知して `要確認` / 判定停止へ送る導線を実装する（`REQ-104`, `REQ-203`, `REQ-304`）。

### 到達基準

- ゴールドスタンダードデータセットで OCR ベンチマークが実施済みである（`REQ-004`）。
- 核家族の基本ケースで、OCR → 構造化抽出 → 相続人候補表示 → 簡易図面生成まで一連実行できる（`REQ-701`）。
- MVP外ケースは誤って自動確定せず、停止理由とレビュー要否を表示できる。

### 判断ゲート（Go/No-Go）

- Go: 主要フィールド正解率 90%以上、MVP内基本ケースの相続判定正解率 95%以上、PoC 範囲で致命的な OCR 波及バグがない。
- No-Go: `RISK-001` または `RISK-007` が未制御で、誤読や判定誤りが下流へそのまま流れる。`RISK-008` に対する回帰評価手段がない。

### 法務・業務監修レビューゲート

- AC-9 必須ゲート: OCR 精度ベンチマークを実施し、ゴールドスタンダードデータセットでフィールド正解率 90%以上を確認する。
- `REQ-601` の対応範囲表示、`REQ-603` の「自動判定は補助である」表示方針を PoC 時点で文案化する。
- 外部 API 利用時の委託整理と越境移転整理は `要確認` とし、Phase 2 進行前に論点一覧を固定する（`OQ-1`, `RISK-004`, `RISK-010`）。

### 対応するMVP外ケース

- 本フェーズでは正式対応しない。以下は検知して停止する。
- `普通養子（実親の相続権維持）`
- `特別養子（実親関係消滅）`
- `数次相続`
- `相続放棄と代襲の関係`
- `相続欠格・推定相続人廃除`
- `認知（死後認知含む）`
- `外国籍相続人`

### 高リスク緩和の重点

- `RISK-001` OCR誤読の相続判定波及: 前処理、信頼度、補正前提の制御を固める。
- `RISK-007` 相続判定ロジックバグ: MVP内ロジックを純粋関数と回帰テストで固定する。
- `RISK-008` OCRモデル更新による精度劣化: ベンチマーク手順とモデルバージョン記録を先に用意する。
- `RISK-002`, `RISK-003`: 名寄せは保守的設定にして誤統合より未統合を優先する。

## Phase 2: β版

- 関連要件: `REQ-202`, `REQ-204`, `REQ-303`, `REQ-304`, `REQ-401`〜`REQ-404`, `REQ-503`, `REQ-702`
- 期間の目安: 8〜12週間

### スコープ

- 案件管理、補正 UI、レビュー承認フロー、PDF/SVG 出力、監査ログを実装する（`REQ-401`〜`REQ-404`, `REQ-303`, `REQ-304`）。
- 市区町村フォーマット差や品質劣化画像への耐性を拡張し、PoC から β 品質へ引き上げる（`REQ-702`）。
- MVP内の例外ケースである代襲、半血兄弟姉妹、胎児、失踪宣告、同時死亡推定、再婚と連れ子、離婚後の子を β 品質で扱えるようにする（`REQ-202`, `REQ-204`）。
- Phase 3 以降で扱う複雑案件に備え、養子種別、認知、放棄、欠格、外国籍、数次相続を検知して停止理由・必要証憑・レビュー導線を出し分ける（`REQ-203`, `REQ-304`）。

### 到達基準

- 複数案件管理、補正後再判定、レビュー承認、PDF/SVG 出力が一連で動作する（`REQ-702`）。
- 司法書士監修用の正解データセット 50 件を投入できる状態になっている。
- 名寄せ精度、補正工数、図面修正率が PoC 比で改善している。

### 判断ゲート（Go/No-Go）

- Go: β 指標として相続判定正解率 95%以上、名寄せ F1 0.93 以上、補正工数 30 分以内、図面修正率 30%以下を満たす。
- Go: 司法書士監修 50 件の差分が分析済みで、重大バグの再発防止策が入っている。
- No-Go: `RISK-002` / `RISK-003` の名寄せ不具合、`RISK-006` のテナント分離欠陥、`RISK-007` の複雑ケース誤判定が未収束。

### 法務・業務監修レビューゲート

- AC-9 必須ゲート: 司法書士による相続判定正解データ 50 件の監修完了。
- `REQ-603` の免責表示、レビュー責任、確定前表示を β UI と PDF に反映する。
- `REQ-602` に基づく外部サービス台帳、保存有無、学習利用有無、越境移転有無を棚卸しする。法的整理未了の項目は `要確認` として Phase 3 へ持ち越し不可とする。

### 対応するMVP外ケース

- βでも正式対応は開始しない。以下は「検知して停止」または「要専門家レビュー」へ送る。
- `普通養子（実親の相続権維持）`
- `特別養子（実親関係消滅）`
- `数次相続`
- `相続放棄と代襲の関係`
- `相続欠格・推定相続人廃除`
- `認知（死後認知含む）`
- `外国籍相続人`
- ただし、停止理由、必要証憑、解除条件、レビュー導線は β で実装する。

### 高リスク緩和の重点

- `RISK-002` 名寄せ False Positive と `RISK-003` False Negative: 候補提示 UI とレビュー承認で抑制する。
- `RISK-006` テナント分離欠陥: RBAC、案件単位制御、監査ログを β で固める。
- `RISK-007` 相続判定ロジックバグ: MVP内例外ケースの回帰テストを拡張する。
- `RISK-001`, `RISK-008`: PoC 指標を継続監視し、帳票差分での劣化を検知する。

## Phase 3: 士業限定運用

- 関連要件: `REQ-202`, `REQ-203`, `REQ-303`, `REQ-304`, `REQ-503`, `REQ-504`, `REQ-602`, `REQ-603`, `REQ-703`
- 期間の目安: 8〜10週間

### スコープ

- 司法書士・行政書士の限定顧客に対し、実案件ベースで運用し、補正工数、レビュー時間、図面修正率、要確認停止率を継続計測する（`REQ-703`）。
- 利用規約、免責事項、プライバシーポリシー、外部 API 利用条件、保持期間ポリシーを契約・運用へ落とし込む（`REQ-504`, `REQ-602`, `REQ-603`）。
- βで停止していた複雑案件のうち、士業監修前提で以下を限定対応する。
- `認知（死後認知含む）`
- `相続放棄と代襲の関係`
- `相続欠格・推定相続人廃除`

### 到達基準

- 少数の士業事務所で継続利用でき、β 指標より改善した KPI が確認できる（`REQ-703`）。
- 規約、免責、プライバシー、保持期間、外部 API 管理についてレビュー可能な文書一式が揃う。
- 複雑案件でも自動確定ではなく、停止・限定確定・専門家レビューの境界が明文化されている。

### 判断ゲート（Go/No-Go）

- Go: 相続判定正解率 97%以上、名寄せ F1 0.95 以上、補正工数 15 分以内、図面修正率 15%以下、重大インシデント 0 件。
- Go: `RISK-004`, `RISK-005`, `RISK-006`, `RISK-010` に対する運用統制が有効に機能している。
- No-Go: 契約・規約・プライバシー文書の法務レビュー未完了、または士業限定運用で重大な誤判定・情報管理事故が出ている。

### 法務・業務監修レビューゲート

- AC-9 必須ゲート: 利用規約・免責事項・プライバシーポリシーの法務レビュー完了。
- `REQ-602` に基づく委託/第三者提供/外国にある第三者提供の整理を完了する。未整理事項は `要確認` のまま本番へ進めない。
- `REQ-603` に基づく責任分界とレビュー責任を契約書・画面・出力物へ反映する。

### 対応するMVP外ケース

- 士業限定で正式対応開始:
- `認知（死後認知含む）`
- `相続放棄と代襲の関係`
- `相続欠格・推定相続人廃除`
- 士業限定でも原則停止または個別審査継続:
- `普通養子（実親の相続権維持）`
- `特別養子（実親関係消滅）`
- `数次相続`
- `外国籍相続人`

### 高リスク緩和の重点

- `RISK-004` 外部 API 送信リスク: 契約、台帳、送信最小化、監査ログを運用に載せる。
- `RISK-005` 暗号化鍵管理不備: KMS、鍵ローテーション、復旧演習を整える。
- `RISK-006` テナント分離欠陥: 限定顧客下でアクセス制御の実運用検証を行う。
- `RISK-010` 個人情報の越境移転: 国内完結構成優先とし、越境移転が残る場合は `要確認` のまま公開しない。
- `RISK-009` 法改正未対応: 四半期レビューとルールバージョン管理を開始する。

## Phase 4: 本番展開

- 関連要件: `REQ-504`, `REQ-601`〜`REQ-604`, `REQ-704`
- 期間の目安: 6〜8週間

### スコープ

- 本番投入チェックリストに基づき、セキュリティ、監査、削除ポリシー、責任分界、法務局提出運用確認を完了する（`REQ-704`）。
- 士業限定運用の結果を反映し、一般提供可能な運用・サポート体制を整える。
- 以下の最難関MVP外ケースについて、正式対応または継続停止の最終判断を行う。
- `数次相続`
- `外国籍相続人`

### 到達基準

- 法務局への提出候補資料としての運用品質が確認され、実案件での提出テストを通過している。
- 保持期間、削除、監査、電子証跡拡張方針が運用設計に統合されている（`REQ-504`, `REQ-604`）。
- 法改正監視、モデル劣化監視、レビュー責任分界が本番 SOP に組み込まれている。

### 判断ゲート（Go/No-Go）

- Go: 本番品質として OCR 指標が安定し、相続判定正解率 97%以上、補正工数 10 分以内、図面修正率 10%以下を継続して達成する。
- Go: 法務局提出テスト 3 件以上で受理が確認される。
- No-Go: `RISK-009` の法改正追随体制が未整備、`RISK-010` の越境移転整理が未完了、または `OQ-1` / `OQ-2` / `OQ-3` / `OQ-5` が未解消のまま公開範囲を拡大している。

### 法務・業務監修レビューゲート

- AC-9 必須ゲート: 法務局への実際の提出テストを最低 3 件実施し、受理を確認する。
- `REQ-601` の法的制約表示、`REQ-603` の免責・責任分界、`REQ-604` の監査証跡拡張方針を本番運用文書へ反映する。
- 地域差や登記官運用差は `要確認` とし、受理実績のないパターンは参考表示または士業レビュー必須を維持する（`OQ-5`）。

### 対応するMVP外ケース

- 本番で正式対応判断:
- `普通養子（実親の相続権維持）`
- `特別養子（実親関係消滅）`
- `数次相続`
- `外国籍相続人`
- 本番でも条件付き対応または専門家レビュー前提を維持:
- `認知（死後認知含む）`
- `相続放棄と代襲の関係`
- `相続欠格・推定相続人廃除`

### 高リスク緩和の重点

- `RISK-009` 戸籍法・民法改正への未対応: 施行日管理と回帰テストを本番運用へ組み込む。
- `RISK-010` 個人情報の越境移転: 国内完結構成または説明・同意・管理台帳を本番要件として確定する。
- `RISK-005` 鍵管理、`RISK-006` テナント分離、`RISK-004` 外部 API 管理を監査対象として定常運用へ移す。
- `RISK-001`〜`RISK-008` は SLO 監視とレビューキュー運用で継続抑制する。
