# 相続関係説明図 自動作成アプリ - 要件定義プロンプト
## Claude Code版 / 技術選定 2026年基準

## 0. この版の前提
このプロンプトは、相続関係説明図アプリの要件定義を **Claude Code** で進めるためのものです。  
元の要件定義プロンプトの骨子は維持しつつ、以下の点を更新しています。

- 技術選定は **2026年時点で実用可能な技術** を前提にする
- 成果物は、会話出力だけでなく **リポジトリ内の Markdown 文書** として作成する
- Claude Code の運用を踏まえ、**既存資料の読解 -> 前提整理 -> 要件化 -> ファイル出力** の流れで進める
- 実装より前に、**要件定義・リスク・ロードマップ・未決事項** を固める

## 1. あなたの役割
あなたは、日本の相続法務・戸籍制度に精通した **シニアプロダクトマネージャー兼システムアーキテクト** です。  
加えて、Claude Code 上でリポジトリを調査し、ファイルを作成・更新しながら、実装着手前に固めるべき要件を網羅的かつ構造的に定義してください。

## 2. Claude Code としての作業方針
以下を厳守してください。

1. まず既存ファイル・既存メモ・入力資料を確認し、既存前提を要約する
2. 情報不足があっても、作業停止せず、合理的な仮定を明示して進める
3. 重要な確認事項だけを最大 3 件まで列挙し、それ以外は `仮定` として前進する
4. いきなり実装コードを書かず、先に `docs/` 以下の要件成果物を作成する
5. 法務・実務運用・法務局受理実務に関わる事項は、未確認なら `要確認` と明示する
6. 各要件はできるだけ「要件ID / 優先度 / 要件内容 / 背景 / 受入基準 / 依存関係」を持たせる
7. 技術選定は、性能だけでなく **説明可能性・監査性・再現性・運用保守性・個人情報保護・コスト** を含めて比較する
8. OCR / LLM / ルールベース判定の責務分離を明確にし、人間補正フローを前提に設計する

## 3. アプリ構想

### 概要
戸籍謄本・改製原戸籍・除籍謄本を OCR や画像解析で読み取り、記載内容（人物、身分事項、続柄、生年月日、死亡日等）を構造化データとして抽出し、法定相続人を自動判定した上で、法務局提出にも使える「相続関係説明図」を自動生成する Web アプリケーション。

### 想定ユーザー
- 司法書士・行政書士（主要ターゲット）
- 相続手続きを自力で進める一般個人
- 士業事務所のスタッフ

### 対象書類
- 戸籍謄本（現行・コンピュータ化後）
- 改製原戸籍（昭和改製・平成改製）
- 除籍謄本
- ※ 手書き戸籍（昭和改製前）の対応可否も検討対象

## 4. 要件定義で求めること
以下のカテゴリごとに、漏れなく・矛盾なく・優先度付きで要件を定義してください。

---

### 4-1. 入力・OCR要件
- 対応すべき書類フォーマットの一覧と優先度
- 手書き戸籍（旧字体・変体仮名・縦書き）への対応方針
- 市区町村ごとのフォーマット差異への対処方針
- OCR 技術の選定基準（精度・コスト・速度・説明可能性・再学習容易性）
- 画像前処理の要件（傾き補正、ノイズ除去、余白検出、ページ分離、縦書き対応）
- OCR 結果に対する人間補正フローの要否と設計
- 画像品質の最低要件（解像度・傾き・折り目・影・かすれ等への耐性）
- 2026年時点の実用技術を前提とした、LLM / マルチモーダルモデル / ルールベース OCR の役割分担
- 評価データセット、精度指標、誤読の重大度分類

### 4-2. データモデル要件
- Person エンティティの属性定義（氏名、旧字体・異体字、生年月日、死亡日、本籍地、最後の住所、登記簿上住所、性別記載、備考等）
- Event エンティティの種類と属性（出生、婚姻、離婚、養子縁組、離縁、転籍、分籍、死亡、失踪宣告、認知 等）
- Relationship エンティティの種類と属性（実親子、養親子、配偶者、前配偶者、兄弟姉妹、半血関係 等）
- 時系列管理（婚姻 -> 離婚 -> 再婚、養子縁組 -> 離縁 等）
- 複数戸籍間の名寄せロジック
- 氏名変更・旧字体差・生年月日表記差・本籍移転を含む同一人物同定
- 戸籍の連続性検証（出生から死亡まで途切れなく追えるか）
- 不確実性管理（候補一致、保留、一致不可の状態管理）

### 4-3. 相続判定ロジック要件
以下のすべてのケースを網羅し、判定ルールとエッジケースを明示してください。

- 法定相続人の確定（配偶者 + 第1〜3順位の判定フロー）
- 代襲相続・再代襲相続の発生条件と範囲
- 養子の扱い（普通養子 / 特別養子で実親との関係がどう変わるか）
- 半血兄弟姉妹の相続分計算
- 相続放棄者の除外処理
- 相続欠格・推定相続人の廃除
- 数次相続（相続人が被相続人の死後に死亡）の対応
- 胎児の相続権
- 行方不明者（失踪宣告）の扱い
- 国際結婚・海外在住者が絡むケースの対応範囲
- 法定相続分の自動計算の要否
- 判定不能時の扱い（自動確定しない条件、要レビュー条件、出力停止条件）

### 4-4. 出力・図面要件
- 相続関係説明図の記載必須事項（法務局提出用の様式・実務慣行への準拠）
- 被相続人の情報：氏名・最後の本籍・最後の住所・登記簿上の住所・死亡日
- 相続人の情報：氏名・住所・生年月日・被相続人との続柄
- 「相続」「分割」「放棄」等の注記ルール
- 家系図ツリーの自動レイアウトアルゴリズム要件
- 世代の段組み
- 配偶者の横並び配置
- 離婚再婚・養子縁組が重なる複雑ケースでのレイアウト破綻防止
- 出力フォーマット（PDF・印刷用A4/A3・画面表示用 SVG / Canvas 等）
- 手動でのレイアウト微調整機能の要否
- 出力物における「自動生成」「要確認」表示の方針

### 4-5. UX・業務フロー要件
- 一つの相続案件の典型的ワークフロー（アップロード -> OCR -> 確認 / 補正 -> 相続人確定 -> 図面生成 -> 出力）
- OCR 結果の確認・補正 UI の設計方針（原本画像と OCR 結果の対照表示、信頼度表示、差分表示 等）
- 複数案件の管理機能の要否
- 案件の保存・再編集・バージョン管理
- 操作ログ・監査証跡の要否（士業の業務品質管理用）
- レビュー待ち・法務確認待ち・顧客確認待ち等の状態管理
- 一般個人向けと士業向けの UX 差分の要否

### 4-6. 技術アーキテクチャ要件
- フロントエンド技術選定
- バックエンド技術選定
- OCR パイプライン構成（前処理 -> OCR / 画像理解 -> 後処理 -> 構造化パース -> 整合性検証）
- LLM 活用箇所の特定（構造化パース、名寄せ、異常検知、説明生成 等）
- ルールエンジン活用箇所の特定（相続判定、必須項目検証、出力制御 等）
- データベース設計方針（RDB / グラフDB / ドキュメントDB / 検索基盤）
- ファイルストレージ（アップロード画像・生成PDF・中間成果物）
- セキュリティ要件（戸籍は最高レベルの個人情報）
- 暗号化（保存時・通信時・鍵管理）
- アクセス制御（RBAC / 監査ログ / テナント分離）
- データ保持期間・削除ポリシー
- オンプレ / プライベートクラウド / パブリッククラウドの判断基準
- 2026年時点での実装現実性、採用難易度、保守性、コスト、ベンダーロックイン比較
- 推奨スタックと代替案を比較表で提示

### 4-7. 法的・コンプライアンス要件
- 個人情報保護法への準拠事項
- 戸籍情報の取り扱いに関する法的制約
- 士業の守秘義務との整合性
- 免責事項・利用規約で明示すべき事項（自動判定の限界等）
- 電子署名・タイムスタンプの要否
- 監査証跡の保存方針
- AI 利用時の説明責任とレビュー責任分界
- 外部API / SaaS 利用時のデータ移転・委託管理上の留意点

### 4-8. 開発フェーズ分割
以下の観点で最適なフェーズ分割を提案してください。

- 各フェーズのスコープとゴール
- 技術的リスクが最も高い箇所の早期検証計画
- MVP（最小実用プロダクト）の定義
- フェーズ間の依存関係
- PoC -> β版 -> 士業限定運用 -> 本番展開の移行条件
- 品質評価指標（OCR精度、相続判定精度、補正工数、図面修正率 等）

## 5. 出力ファイル
Claude Code では、会話回答だけで終わらせず、以下のファイルを作成または更新してください。

1. `docs/requirements-definition.md`
   - 上記 8 カテゴリを構造化
   - 各要件に優先度 [Must / Should / Could]
   - 可能なら要件ID・背景・受入基準・依存関係も付与

2. `docs/edge-cases.md`
   - 相続判定で特に注意が必要なパターンを表形式で列挙
   - 各行に「ケース / 何が難しいか / 判定上の注意点 / 実装上の注意点 / レビュー要否」を持たせる

3. `docs/tech-risk-register.md`
   - リスク項目 / 影響度 / 発生可能性 / 早期検知方法 / 対策案 / 代替策

4. `docs/roadmap.md`
   - 推奨フェーズ計画をロードマップ形式で整理
   - 各フェーズの到達基準・判断ゲートを明記

5. `docs/open-questions.md`
   - ステークホルダー確認が必要な判断事項
   - 「論点 / なぜ未決か / 誰が決めるか / 決定期限 / 影響範囲」で整理

## 6. 制約・前提
- 日本国内の相続法（民法）に準拠すること
- 戸籍法・戸籍法施行規則の関連条文を根拠として明示すること
- 令和6年4月1日施行の戸籍法改正（広域交付制度等）も考慮すること
- 実務上の慣行（法務局の実際の受理基準等）も可能な限り反映すること
- 技術選定は **2026年時点で実用可能なもの** に限定すること
- 法令・実務運用・審査運用が未確認の事項は、推測で埋めず `要確認` と明示すること

## 7. 期待する品質
- 漏れなく、矛盾なく、優先度付きで整理されている
- 業務要件、法務要件、データ要件、技術要件が分離されている
- 実装チームがそのまま設計着手できる粒度まで具体化されている
- 自動化できる範囲と、人間レビュー必須の範囲が分離されている
- 不確実性と未決事項が明示されている
- 将来の要件追加（遺言、遺産分割協議書連携、登記申請補助等）に拡張しやすい

## 8. 実行時の進め方
作業開始時は次の順序で進めてください。

1. 既存資料を確認して前提・制約・不足情報を短く要約する
2. 必要なら重要確認事項を最大 3 件まで列挙する
3. その時点で置く仮定を明示する
4. `docs/` 以下の 5 ファイルを作成 / 更新する
5. 最後に、未解決リスクと追加調査項目を簡潔にまとめる
